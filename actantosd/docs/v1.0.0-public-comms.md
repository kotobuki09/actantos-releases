### Problem

Coding agents and MCP tools can read secrets, run shell, and push code with little runtime oversight. Prompt filters and post-hoc dashboards don’t block the tool call. Teams need a **deterministic gate** between the agent and the world.

### Solution

ActantOS sits **in-path and out-of-process**. Before every tool call:

1. Intercept the request  
2. Evaluate Cedar policy + risk rules + budgets + kill switch  
3. Return **allow**, **deny**, or **approval_required**  
4. Only then may execution proceed—with a bound decision token and audit evidence  

### How it works (simple diagram copy)

```text
Agent (Cursor / Pi / MCP client)
        │
        ▼
  ActantOS actantosd  ──►  Decision: allow | deny | approval_required
        │                         │
        │ allow                   │ approval_required → human (web/Slack)
        ▼                         ▼
  Docker sandbox / upstream tools
        │
        ▼
  Tool result + hash-chained audit
```

# 5. Website — FAQ

**What is ActantOS?**  
A runtime control plane that authorizes agent tool calls before they execute.

**Is v1 free?**  
Yes for self-host open-core (kernel + operator UI + optional Slack). Hosted enterprise is separate.

**Do I need MCP?**  
No. Coding pilots use the **Pi Primary Path**. MCP is optional.

**Does it use an LLM to approve actions?**  
No. Authorization is deterministic (Cedar + rules + state), not LLM-as-judge.

**What happens if ActantOS is down?**  
Fail-closed: adapters do not execute tools without an allow decision.

**Can I upgrade from pilot beta?**  
Yes—see upgrade guide `v0.7` → `v1.0.0` (`npm run db:migrate`).

# 10. Technical detail (docs site / GitHub wiki)

### Runtime surface (stable)

| Method | Path | Role |
| --- | --- | --- |
| `POST` | `/v1/intercept/tool-call` | Decision before execute |
| `POST` | `/v1/tool-result` | Result + decision_token verify |
| `GET` | `/v1/mcp/sse` | MCP gateway transport |
| `POST` | `/v1/mcp/message` | MCP messages |

Operator surfaces include agents, sessions, decisions, approvals, kill switches, evidence export, policy bundles, budgets, rate limits, risk rules, usage metrics (Postgres for full operator set).

### Decision outcomes

- `allow` — may execute (token issued)  
- `deny` — must not execute  
- `approval_required` — human one-use grant required  

### Default coding posture (Balanced)

- Workspace read/write: allow (non-credential)  
- Secrets / credential paths: deny  
- `git push` / publish-class / dangerous network: approval_required  

### Stack

- TypeScript, Fastify, Zod, Postgres, Kysely  
- Cedar policy engine  
- Docker Compose for self-host  
- Default daemon port `3100`  

### Verification commands

```bash
npm run release:verify
npm run smoke:fresh-install
npm run release:artifacts
```
